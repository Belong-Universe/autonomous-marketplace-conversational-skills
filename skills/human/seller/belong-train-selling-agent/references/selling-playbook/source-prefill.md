# Source Prefill

[Back to index](index.md) | Next: [Value Proposition](01-value-proposition.md)

Use this before drafting section content. The goal is to reduce manual questioning by mining existing seller material, then still review the result section by section.

## Sources To Request

Ask whether the seller has any of these:

- Drive folder, shared folder, or data lake
- Company website, service page, product docs, help center, or blog
- Sales deck, one-pager, proposal archive, pricing sheet, case studies, or CRM export
- Legal templates, MSA, SOW, order form, DPA, NDA, security terms, or procurement notes
- Delivery SOPs, onboarding checklists, project plans, status templates, QA checklists, or acceptance templates
- Support history, dispute history, refund notes, escalation logs, meeting notes, or postmortems

## Prefill Map

Build a compact map before asking section questions.

| Source | Sections informed | Confidence | Gaps |
| --- | --- | --- | --- |
| Source name | Section names | High, Medium, Low | Short missing list |

Use `High` when the source directly states policy or facts. Use `Medium` when the source implies a rule but needs confirmation. Use `Low` when the source only hints at a rule.

## Extraction Rules

- Extract facts and operating rules, not marketing filler.
- Preserve numbers, thresholds, deadlines, dates, payment terms, and approval limits exactly.
- Keep source-backed details separate from assumptions.
- If two sources conflict, record both and ask the human to choose.
- If access fails, state the problem and ask for pasted excerpts or another source.
- If a source helps a future section, store that fact for later but do not skip approval gates.

## Output Before Review

Before moving into [Value Proposition](01-value-proposition.md), show:

- Sources accessed
- Sources unavailable
- Initial section coverage
- Highest-confidence prefilled sections
- Biggest gaps
- Request to begin section-by-section review
